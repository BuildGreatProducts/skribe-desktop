## Build Philosophy

1. **Each phase ships something usable.** No phase leaves the app broken. After any phase, you can launch `Skribe.app` and see something functional.
2. **Infrastructure before features.** Phase 0 sets up the Tauri shell, the design system, and the persistence layer so feature work in Phases 1–3 is fast and clean.
3. **Magic moment first.** The "agent that already knows your project" demo must be working by the end of Phase 2. Everything in Phase 3 is polish on top of a working magic moment.
4. **Test as you go.** Every task includes a verification step. Don't save testing for Phase 3.
5. **Progressive enhancement.** Start with the simplest working version of each surface, then layer on polish. The Tiptap editor in Phase 1 doesn't need every formatting option on day one.
6. **Review before proceeding.** Push each completed phase as a PR and let a review agent (e.g. [CodeRabbit](https://coderabbit.ai)) review it before starting the next phase. External review catches issues the coding agent won't flag.
7. **Beauty as you go.** Skribe is a designer-led product for a design-aware audience. Don't defer typography, spacing, or color decisions to Phase 3 — they ship correctly in Phase 0 and propagate forward.
8. **Local-first, on-disk, always.** No phase introduces cloud dependencies, telemetry, or hidden state. Every file the user sees is plain markdown on disk; every setting is plain JSON in `~/Library/Application Support/Skribe/`.

## Agent Session Guide

### How to Use This Roadmap with Your Coding Agent

1. **Start a session:** Give your coding agent the phase prompt at the beginning of each phase. The phase prompt is at the top of each phase block above.
2. **Read selectively:** Each phase lists its Reference sections — the specific parts of the PRD, vision doc, and GTM doc needed for that phase. The agent should read only those sections, not the entire documents. Loading the full documents wastes context.
3. **Let it work:** The agent reads the roadmap, finds the first unchecked task, implements it, and marks it complete by changing `- [x]` to `- [x]` in this file.
4. **One session = one phase (ideally):** Try to complete a full phase in one session for best continuity. If you need to stop, the agent can resume from the last unchecked task.
5. **Push a PR for review:** When a phase is complete, push the work as a PR and let a review agent (e.g. [CodeRabbit](https://coderabbit.ai)) review it before starting the next phase. See the Phase Review section below.
6. **Need more context?** If a task references a section not in the phase's Reference sections, the agent should read just that section on demand.

### Session Tips

- **Don't read everything:** The PRD, vision doc, and GTM doc are large. Each phase's Reference sections tell the agent exactly what to read.
- **Don't skip tasks:** Tasks are ordered intentionally. Skipping creates dependency issues.
- **Verify after each phase:** Run `npm run tauri dev` after completing a phase to confirm everything works before moving on.
- **Verify the magic moment after Phase 2:** TASK-045 is the most important verification in the entire roadmap. Don't move to Phase 3 until the "agent that already knows your project" demo works reliably on a real folder.
- **Review before moving on:** Push a PR for each completed phase and let your review agent check it. Don't start the next phase until the PR is merged.
- **Update the status line:** After completing tasks, update the header status: `**Status:** X/56 tasks complete` and `**Current Phase:** Phase N`.

### Prompt Templates

**Starting a new phase:**
> "Read docs/product-roadmap.md and find the current phase. Read only the Reference sections listed for that phase from docs/prd.md, docs/product-vision.md, and docs/gtm.md. Start working on the first unchecked task. After completing each task, update the checkbox to [x] in the roadmap file. Continue through the phase."

**Resuming after a break:**
> "Read docs/product-roadmap.md. Find where we left off (first unchecked task). Read only the Reference sections listed for the current phase. Continue from the first unchecked task."

**After completing a phase:**
> "Phase [N] is complete. Create a branch called phase-{N}/{slug}, commit all work, push, and open a PR targeting main. Title it 'Phase {N}: {Title}' and include the phase goal and completed task count in the body."

**Fixing an issue:**
> "There's a problem with [description]. Read the relevant section of docs/prd.md for the expected behavior and fix it. Don't mark any new tasks complete until the fix is verified."

### Phase Review Workflow

After every phase is complete, push the work as a pull request before starting the next phase. This is the quality gate.

1. **Branch:** `phase-{N}/{phase-slug}` (e.g. `phase-0/foundation-and-setup`).
2. **Commit & push:** Stage all phase work, commit with the message `Phase {N}: {Phase Title}`, and push to origin.
3. **Open a PR:**
   - Title: `Phase {N}: {Phase Title}`
   - Body: phase goal, number of tasks completed, what to verify manually, and the Reference sections used.
4. **Review:** Let your review agent (CodeRabbit or your tool of choice) review the PR. If you don't have one, [CodeRabbit](https://coderabbit.ai) is free for open-source.
5. **Address feedback:** If the review agent leaves comments, fix them before merging. Update task checkboxes only after fixes are applied.
6. **Merge:** Once review passes, merge the PR. Then proceed to the next phase.
